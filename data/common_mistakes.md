# Типичные ошибки при составлении ТЗ

## 1. Отсутствие бизнес-контекста

**Проблема:** Заказчик описывает ЧТО хочет сделать, но не объясняет ЗАЧЕМ.

**Пример плохого ТЗ:**
> Нужен лендинг с формой обратной связи

**Почему это плохо:**
- Непонятно, для какого продукта/услуги
- Неизвестна целевая аудитория
- Нет критериев успеха

**Правильный подход:**
Спросить: "Какую задачу должен решать этот лендинг? Что должен сделать посетитель?"


## 2. Размытые формулировки

**Опасные слова:**
- "Красивый"
- "Современный"  
- "Удобный"
- "Интуитивный"
- "Качественный"

Все эти слова субъективны и у каждого человека означают разное.

**Решение:** Просить референсы, примеры, конкретные метрики.


## 3. Scope creep (расползание объёма)

**Признаки:**
- "А можно ещё добавить..."
- "Это же мелочь, займёт 5 минут"
- "Я думал, это входит по умолчанию"

**Профилактика:**
- Чётко фиксировать границы в ТЗ
- Любые дополнения — через change request
- Оценивать влияние на сроки и бюджет


## 4. Игнорирование технических ограничений

**Примеры:**
- Заказчик хочет интеграцию с системой, которая не имеет API
- Требуется функционал, невозможный на выбранной платформе
- Ожидания по производительности нереалистичны

**Решение:** Техническая экспертиза до начала работ.


## 5. Отсутствие приоритетов

**Проблема:** Всё одинаково важно = ничего не важно.

Когда нет приоритетов:
- Невозможно сократить scope при необходимости
- Ресурсы распыляются
- Критичные функции могут не попасть в релиз

**Решение:** Разделить функционал на Must Have / Should Have / Could Have.


## 6. Нереалистичные сроки

**Типичные причины:**
- "Нужно вчера"
- Привязка к внешнему событию без запаса времени
- Недооценка сложности

**Последствия:**
- Низкое качество
- Выгорание команды
- Технический долг

**Решение:** Добавлять буфер 20-30% к оценкам.


## 7. Один человек = много ролей

**Проблема:** Заказчик одновременно:
- Принимает решения
- Даёт обратную связь
- Тестирует
- Согласовывает бюджет

Это создаёт bottleneck и затягивает проект.

**Решение:** Определить RACI-матрицу до старта.


## 8. Недостаточная детализация UX

**Примеры пропущенных моментов:**
- Что происходит при ошибке?
- Как выглядит пустое состояние?
- Какие есть уведомления?
- Как работает поиск при нулевых результатах?

**Решение:** Проработать edge cases и error states.
